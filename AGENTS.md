## Контекст проекта

Проект развивается в сторону portfolio MVP: **Fridge-to-Meal Planner AI**.
Цель - показать практичную AI-систему, которая помогает собрать меню из продуктов дома, учитывает бюджет, сроки годности, сезонность и цели по КБЖУ, а также формирует список покупок.

Текущий стек:

- FastAPI
- Pydantic
- статический HTML/CSS/JS frontend
- JSON-база рецептов
- optional computer vision через YOLO/OpenCV

## Правила работы с кодом

- Сначала изучай текущую структуру и существующую логику.
- Не переписывай проект с нуля без явной причины.
- Делай минимальные безопасные изменения.
- Не ломай существующие API `/api/v2/suggest` и `/api/v2/plan` без отдельного решения.
- Для MVP приоритет у надежного pantry/meal-planning flow, а не у дорогого computer vision.
- Vision-функции должны быть optional и всегда требовать подтверждения пользователя перед добавлением продуктов.

## Секреты и локальные артефакты

- Никогда не выводи значения из `.env`.
- Не добавляй в git `.env`, `.venv`, `__pycache__`, локальные модели и веса `*.pt`.
- Для примеров конфигурации используй только заглушки и `.env.example`.
- Не проси пользователя вставлять реальные API-ключи, токены, пароли или database URL.

## Тестирование и проверка

- После backend-изменений проверяй FastAPI endpoints через pytest или `curl`/Swagger.
- После frontend-изменений проверяй основной demo flow: pantry -> preferences -> meal plan -> shopping list/statistics.
- После изменений в зависимостях проверяй установку в чистом окружении.
- После изменений упаковки проекта проверяй, что `.env`, `.venv` и модели игнорируются git.

## Production-подход

- Учитывай таймауты, валидацию входных данных, CORS, лимиты upload-файлов, логирование и безопасные defaults.
- PostgreSQL, Redis, Alembic и RAG добавлять только когда появляется реальный код, который их использует.
- Не выполнять опасные действия вроде удаления файлов, сброса БД, force push или production-миграций без явного подтверждения.
