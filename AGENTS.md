## Контекст проекта

Проект развивается в сторону **Food Operations Copilot**: human-controlled системы, которая снимает
ежедневную нагрузку выбора еды и покупок, но не принимает решения вместо человека.

Ключевой принцип:

> AI может наблюдать, предлагать, ранжировать и объяснять. Подтверждённые факты и итоговые решения всегда
> принадлежат пользователю.

Текущий стек:

- FastAPI и Pydantic;
- модульный backend в `app/`;
- статический demo frontend;
- JSON-каталог рецептов;
- optional CV через отдельный experimental-контур.

## Архитектурные правила

- Сначала изучай существующую логику и публичные API.
- Не ломай `/api/v2/*`: это compatibility API существующего demo.
- Новые human-controlled контракты добавляй в `/api/v3/*`.
- HTTP-логику размещай в `app/api`, доменную логику - в `app/services`.
- Не добавляй PostgreSQL, Redis, Alembic или внешнюю модель без реального использующего их кода.
- Не скрывай бизнес-решения внутри LLM prompt или model weights.
- Любое предложение AI должно содержать источник, confidence или объяснение.

## Human Override

- Vision/OCR/VLM создают только кандидатов и никогда автоматически не меняют pantry.
- Интерпретация контекста создаёт предлагаемые ограничения, требующие подтверждения.
- План и список покупок остаются черновиками до явного approval.
- Будущие approval/override события должны быть append-only и содержать actor, reason, timestamp,
  policy/model version и исходное предложение.

## Секреты и приватные данные

- Никогда не выводи значения из `.env`.
- Не добавляй в git `.env`, `.venv`, `__pycache__`, локальные БД, фото, чеки, модели и веса `*.pt`.
- Используй `.env.example` только с безопасными заглушками.
- Пользовательские фото и чеки считай приватными; сохранять их для обучения можно только через явный opt-in.

## Проверки

После backend-изменений запускай:

```powershell
.\.venv\Scripts\ruff.exe check app tests run_ultra_smart_app.py run_local_demo.py
.\.venv\Scripts\black.exe --check app tests run_ultra_smart_app.py run_local_demo.py
.\.venv\Scripts\python.exe -m pytest -q
```

После изменения API проверяй:

- compatibility `/api/v2/*`;
- `needs_confirmation=true` для vision;
- `requires_approval=true` для v3 draft-планов;
- валидацию входных данных и upload limits;
- отсутствие секретов и локальных артефактов в git.

## Production-подход

- Используй безопасные defaults, таймауты, лимиты, request ID и структурированные логи.
- Не выполняй удаление данных, force push, production-миграции и изменение production-конфигов без явного
  подтверждения.
- Тяжёлые CV/LLM зависимости должны быть optional; их отказ не должен ломать основной planner.
- Любая модель должна иметь версию, метрики, fallback и понятную границу полномочий.
